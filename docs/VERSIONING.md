# نظام الإصدار الموحّد (Cache Versioning)

## المشكلة اللي بيحلها هذا النظام

قبل كده، كل مكان في الموقع كان بيحتفظ برقم إصدار (`?v=`) منفصل — بعضها كان بيتحدّث
يدويًا، بعضها اتنسي لمدة أيام، وبعض ملفات الجافاسكريبت مكنش عندها رقم إصدار خالص.
النتيجة: تعديلات فعلية على الكود كانت بتوصل للسيرفر لكن متظهرش لبعض الزوّار، لأن
متصفحهم كان محتفظ بنسخة قديمة مخزّنة (كاش) ومفيش حاجة تجبره ياخد الجديد.

الحل: **مصدر واحد للإصدار (Single Source of Truth)**، بيتولّد ويتحدّث تلقائيًا —
محدش (لا إنت ولا أي جلسة Claude مستقبلية) محتاج يعدّله بإيده تاني.

## ماذا يحدث عند كل commit

في كل مرة تُعمل فيها `git commit` (بأي أداة — سطر الأوامر، GitHub Desktop، VS Code،
أو Claude)، بيحصل تلقائيًا وبشكل خفي:

1. بيتولّد رقم إصدار جديد من التاريخ والوقت الحاليين (مثال: `v20260718122448`).
2. نفس الرقم ده يتكتب في **كل** الأماكن دفعة واحدة:
   - `sw.js` → `CACHE_VER`
   - `version.js` (ملف جديد) → `window.APP_VERSION` — تقدر تتأكد بأي وقت من نسخة
     الموقع اللي شغّالة فعليًا في متصفحك بفتح الـ Console وكتابة `APP_VERSION`.
   - `manifest.json` → حقل `"version"` (معلوماتي فقط)
   - كل ملفات `*.html` المتتبَّعة في git → كل `?v=` على ملفات `.js`/`.css`/رابط
     المانيفست، **وبيضيف `?v=` تلقائيًا لأي سكريبت أو ستايل محلي ملوش رقم إصدار
     خالص** (كان ده سبب جزء من المشكلة القديمة).
3. كل الملفات دي تتضاف تلقائيًا (`git add`) **لنفس الـ commit** اللي إنت بتعمله —
   مفيش commit إضافي منفصل، الرقم بيندمج في نفس اللحظة.
4. لو حصل أي خطأ غير متوقع أثناء الخطوات دي (مثلاً PowerShell مش موجود على الجهاز،
   أو أي مشكلة تانية) — **الـ commit بيكمل عادي زي ما هو، من غير ما يتوقف أبدًا**،
   وبس بتظهر رسالة تحذير واضحة تقولك تشغّل السكريبت يدويًا قبل الرفع القادم.

**مفيش حاجة تانية تحتاج تعملها.** كمّل commit وpush زي ما بتعمل بالظبط.

## أين يوجد الـ Hook

- **`githooks/pre-commit`** — السكريبت اللي Git بيشغّله تلقائيًا قبل كل commit
  (ملف متتبَّع في git، مش مخفي جوه `.git/`).
- **`bump-version.ps1`** — السكريبت الفعلي (PowerShell) اللي بيعمل كل التحديثات
  المذكورة فوق. `githooks/pre-commit` بس بينادي عليه.
- الربط بينهم بيتم عبر إعداد Git محلي: `core.hooksPath = githooks` (بدل المسار
  الافتراضي `.git/hooks` اللي مش متتبَّع في git أصلًا).

## لو احتجت تشغّله يدويًا

مش لازم في السير العادي، لكن لو حبيت تتأكد بنفسك في أي وقت:

```
./bump-version.ps1
```

## كيف تُثبّته على جهاز جديد أو لمطوّر جديد

إعداد Git `core.hooksPath` محلي بطبيعته (مش بيتنسخ تلقائيًا مع `git clone`).
عشان تُفعّله على أي جهاز جديد (أو لو انضم مطوّر جديد للمشروع)، خطوة واحدة بس،
تتعمل مرة واحدة بعد الـ clone:

```
cd Makani-spot
git config core.hooksPath githooks
```

بعد الأمر ده، كل commit من على الجهاز ده هيفعّل النظام تلقائيًا زي ما هو موضّح فوق.

## ملاحظات

- لو عملت `git commit --no-verify`، الـ hook هيتخطّى بالكامل (سلوك Git القياسي
  المقصود) — مفيش تحديث إصدار هيحصل في الحالة دي.
- الـ hook بيلمس بس الملفات اللي فيها رقم الإصدار (`sw.js`, `version.js`,
  `manifest.json`, وسطور `?v=` جوه ملفات HTML) — مبيغيّرش أي محتوى تاني.
- `style.css` بقى بياخد نفس الرقم الموحّد زي أي ملف تاني — مفيش تحديث يدوي منفصل
  ليه بعد النهاردة.
