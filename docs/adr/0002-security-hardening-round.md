# ADR 0002 — تصليب أمني: RLS دائم-الصحة، UUID مُرمَّز، search_path، pg_net، بوابة p_secret

**الحالة:** مقبول ومُنفَّذ جزئيًا (2026-07-05) — بند واحد موثَّق ومؤجَّل بقرار، بندان يحتاجان قرارًا/إجراءً من المستخدم مباشرة.

## السياق

مراجعة أمنية شاملة لستة عناصر مُبلَّغة + مراجعة `get_advisors(security)` الكاملة. تم التحقق المباشر من كل عنصر عبر القاعدة قبل أي تنفيذ.

## القرارات

### 1. `organizer_profiles` UPDATE/INSERT (`true` → مقيَّد) — نُفِّذ
كانت أي جلسة مسجَّلة (وحتى anon في INSERT) تستطيع تعديل بروفايل أي منظّم بازار آخر عبر طلب API مباشر (كود العميل الفعلي كان يستخدم `.eq('user_id', self)` فقط بالمصادفة، لا بحكم RLS). الشرط الجديد: `(auth.uid() = user_id) OR is_admin()` — مطابق تمامًا لنمط "المنظم يدير بروفايله" (ALL) المستخدَم أصلاً لنفس الجدول. أُثبِت حيًّا: مستخدم عشوائي *false*، صاحب الحساب *true*، الأدمن *true*.

### 2. UUID الأدمن المُرمَّز (`bazaar_abuse_reports`, `bazaar_bookings`) → `is_admin()` — نُفِّذ
الحساب المُرمَّز (`3bebf8f4-...`) دوره الفعلي `admin` حاليًا، فالسلوك الحالي لم يتغيّر — لكن الاعتماد كان هشًا (لا يغطي أي أدمن آخر، وينكسر صامتًا لو تغيّر دور هذا الحساب تحديدًا). أُثبِت حيًّا: نفس الحساب ما زال يمر، مستخدم عادي لا يمر.

### 3. `search_path` غير ثابت لـ44 دالة (كان 58 — انخفض بفعل حذف التكرار في جولة سابقة) — نُفِّذ
39 منها `SECURITY DEFINER` (بينها `is_admin()` نفسها) — ثغرة من فئة Schema Hijacking: تُنفَّذ بصلاحيات مالك الدالة بينما مسار البحث يعتمد على إعداد جلسة المستدعي القابل للتلاعب. القيمة المختارة: `'public'` **وليس** `''` الفارغة — تحققنا أن عدة دوال (`cleanup_expired_bookings`, `owner_update_booking_status`, `gen_payment_invoice_number`) تستخدم أسماء غير مؤهَّلة بالجدول (`bookings` بدل `public.bookings`)؛ تثبيت مسار فارغ كان سيكسرها فعليًا. `'public'` يطابق التصرّف الافتراضي الحالي تمامًا (صفر تغيير سلوكي) مع تثبيته بحيث لا يتأثر بأي تلاعب مستقبلي بجلسة المستدعي.

### 4. `pg_net` في public → `extensions` — نُفِّذ
تحقق مسبق: صفر دالة داخلية تستدعي أعضاء pg_net، صفر كائن تابع، صفر طلب معلَّق، سجل استجابة واحد تافه. الإضافة لا تدعم `ALTER EXTENSION SET SCHEMA` مباشرة (خطأ Postgres 0A000) — استُخدم الحل الموثَّق رسميًا من Supabase: `DROP EXTENSION` ثم `CREATE EXTENSION ... SCHEMA extensions`.

### 5. بوابة `p_secret` الشكلية — **موثَّقة، لم يُنفَّذ تغيير في التوقيعات**
اكتُشف أن آلية الدخول الفعلية لصفحات الأدمن **ليست** sha256 كما وُصفت — بل `supabase.auth.signInWithPassword()` حقيقي (أُصلح في جولة سابقة). `p_secret` معامل متبقٍّ فقط، يُمرَّر بقيمة ثابتة (`'authenticated'` أو `''`) من كل صفحات الأدمن إلى 37 دالة RPC — تحقُّق SQL مباشر أثبت أن **لا دالة واحدة من الـ37 تتحقق منه فعليًا**؛ الحماية الحقيقية الوحيدة `is_admin()`.

**القرار: توثيق فقط، بلا تعديل توقيعات الدوال الآن.** السبب: جعله اختياريًا (`DEFAULT NULL`) يتطلب `CREATE OR REPLACE FUNCTION` لكل الـ37 دالة (Postgres لا يسمح بإضافة Default لمعامل موجود عبر ALTER) — أي إعادة كتابة/نسخ الجسم الكامل لكل دالة، وهو عمل ميكانيكي كبير يحمل خطر خطأ نسخ حقيقي مقابل فائدة تجميلية بحتة (المعامل مؤكَّد عديم الأثر فعليًا الآن). يُقترح تنفيذه في جولة مستقلة مخصَّصة (تعديل الدوال + تنظيف كل صفحات الأدمن السبع معًا)، لا ضمن جولة تصليب أمني.

### 6. حماية كلمة المرور المسرَّبة — **يتطلب إجراء من المستخدم مباشرة**
إعداد على مستوى خدمة Supabase Auth (GoTrue)، لا تملك أدوات هذه الجلسة وصولًا برمجيًا له. يُفعَّل من: **Supabase Dashboard → Authentication → Policies → Password Security → Leaked password protection**. لا يؤثر على المستخدمين الحاليين أو الجلسات القائمة — يفحص فقط كلمات المرور الجديدة عند التسجيل/التغيير مقابل قاعدة بيانات تسريبات معروفة (نمط HaveIBeenPwned)؛ صفر تعطيل متوقَّع.

## اكتشاف منفصل — غير منفَّذ، بانتظار قرار
`profiles.profiles_public_read_basic` (`SELECT` بشرط `true` لأدوار anon+authenticated): يعرض email/phone/role/is_suspended/subscription_status لكل المستخدمين علنًا. نفس فئة ثغرة (سياسة صحيح-دائمًا) مثل البند 1، اكتُشف ووُثِّق سابقًا في جولتي الأداء وRLS، يُعاد طرحه هنا لأنه يلائم هذه المرحلة، لكن لم يُنفَّذ لأنه خارج النطاق المُصرَّح به صراحةً لهذه الجولة.

## اكتشافات جانبية — مراجَعة ومُصنَّفة "غير قابلة للتنفيذ/غير ضرورية"
- 6 جداول RLS مفعَّل بلا سياسات (`admin_metrics` وغيرها) — رفض افتراضي كامل، مطابق لنمط `admin_audit_log` الموجود، تصميم صحيح مقصود.
- `organizer_notifications` INSERT مفتوح (`true`) — على الأرجح مسار إشعارات لا يتطلب جلسة، Insert-only لا يُسرّب بيانات.
- **اكتُشف عرَضًا (غير أمني):** `cleanup_expired_bookings()` تفشل فعليًا بخطأ `text = uuid` (مقارنة نوع خاطئة، `bz.id::text = bk.bazaar_id` بينما `bazaar_id` من نوع uuid) — خطأ حقيقي موجود مسبقًا، لا علاقة له بهذه المرحلة، غير مجدولة عبر pg_cron حاليًا فأثرها محدود لحين استدعائها يدويًا. موثَّق هنا فقط، لم يُصلَح.
